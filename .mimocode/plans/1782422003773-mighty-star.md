# Plano: Configurações da Plataforma — SQLite-backed

## Contexto

As 3 páginas de settings (Geral, Perfil, Integrações) são hoje apenas placeholders com texto "em breve". O projeto já tem Prisma + SQLite configurado, mas nenhum model de Settings existe. Configurações atuais vêm de `.env` (API keys, provider) e de defaults hardcoded no schema Prisma (model gpt-4o, temperature 0.7, maxTokens 4096).

**Objetivo:** Criar um model `Setting` genérico (key-value) no SQLite, implementar Server Actions para CRUD, e conectar as 3 páginas de settings com forms funcionais.

---

| OPENAI_API_KEY / ANTHROPIC_API_KEY | Idioma padrão |
| WEBHOOK_SECRET | Timezone |
| NODE_ENV | URL de exibição |
| | Provider LLM padrão (openai/anthropic/deepseek) |
| | Modelo padrão para novos agentes |
| | Temperatura padrão |
| | MaxTokens padrão |
| | Threshold de qualificação |
| | Auto-qualificação habilitada |
| | Máx. leads por página |
| | Notificações por email |
| | Política de secrets (allowDatabaseStorage, etc.) |

## Estrutura do Model Setting

```
model Setting {
  id        String   @id @default(cuid())
  key       String   @unique          // ex: "platform.name"
  value     String                    // JSON stringified ou texto plano
  type      String   @default("text") // "text" | "number" | "boolean" | "json" | "secret"
  group     String                    // "general" | "profile" | "integrations" | "agents" | "leads" | "security"
  label     String                    // "Nome da Plataforma"
  helpText  String?                   // texto de ajuda
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

**Por que key-value e não colunas fixas:** Flexibilidade para adicionar configs sem migration. Cada config tem `group` para filtrar por página, `type` para renderizar o input correto, `label` para exibição.

## Fluxo de Implementação

### Passo 1: Schema + Migration
- Adicionar `Setting` model ao schema.prisma
- Rodar `npx prisma db push` (SQLite não usa migrate dev normalmente neste projeto)

### Passo 2: Seed
- Adicionar configurações padrão ao seed.ts
- Inserir todas as configs listadas acima com valores default

### Passo 3: Server Actions + Helpers
- Criar `src/app/actions/settings.ts` com:
  - `getSettings(group?)` — retorna configs por grupo
  - `getSetting(key)` — retorna uma config
  - `updateSetting(key, value)` — atualiza uma config
  - `updateSettings(entries)` — atualiza múltiplas configs
  - `initializeDefaultSettings()` — insere defaults se não existirem
- Criar `src/lib/settings.ts` com:
  - `getSettingValue(key)` — helper rápido para ler valor
  - `getSettingsByGroup(group)` — helper para agrupar

### Passo 4: Página Geral
- Form com seções: Plataforma, Agentes, Leads
- Cada campo lê do banco via Server Action
- Salvar chama updateSettings bulk
- Usar react useState para formulário controlado
- Toast de sucesso/erro com `sonner`

### Passo 5: Página Perfil
- Form com: nome, email, papel, notificações
- Mesmo padrão de Server Actions

### Passo 6: Página Integrações
- Listar integrações do model `Integration` existente
- Cards com status (ativa/inativa)
- Botão para toggle isActive
- Sem form de criação por agora (usar seed ou API)

### Passo 7: Conectar LLM Client
- Modificar `createLLMClient` para ler provider do SQLite
- Fallback para .env se settings não existirem
