# Plano: Corrigir erro `deletedAt` no banco SQLite

## Problema

A tabela `Lead` no banco SQLite foi criada a partir de uma versão anterior do schema que **não possuía** as colunas `deletedAt` e `externalId`. A migration `20260624084500_data_persistence_foundation` foi documentada mas **nunca foi aplicada** (não existe tabela `_prisma_migrations` no banco). O código em `src/app/actions/leads.ts` usa `deletedAt` em 6 consultas Prisma, causando erro `P2022: The column main.Lead.deletedAt does not exist`.

### Causa Raiz

O banco foi criado com `prisma db push` ou `prisma db seed` usando uma versão antiga do schema. A migration formal nunca rodou. O GPT confirmou: "prisma migrate dev e prisma migrate deploy falharam neste ambiente com Schema engine error".

### Colunas faltantes na tabela Lead

| Coluna | Prisma Schema | Banco Atual |
|--------|--------------|-------------|
| `deletedAt` | `DateTime?` | **AUSENTE** |
| `externalId` | `String? @unique` | **AUSENTE** |

### Tabela Integration com nome errado

| Campo Prisma | Banco Atual |
|-------------|-------------|
| `secretRef` | `credentials` |

### Índices faltantes na tabela Lead

- `Lead_status_deletedAt_idx` — AUSENTE
- `Lead_deletedAt_idx` — AUSENTE

## Plano de Execução

### Passo 1: Aplicar SQL direto no banco SQLite

Aplicar as alterações ausentes via `sqlite3`:

```sql
-- Adicionar colunas faltantes na tabela Lead
ALTER TABLE "Lead" ADD COLUMN "deletedAt" DATETIME;
ALTER TABLE "Lead" ADD COLUMN "externalId" TEXT;

-- Criar índice único para externalId
CREATE UNIQUE INDEX "Lead_externalId_key" ON "Lead"("externalId");

-- Criar índices compostos
CREATE INDEX "Lead_status_deletedAt_idx" ON "Lead"("status", "deletedAt");
CREATE INDEX "Lead_deletedAt_idx" ON "Lead"("deletedAt");

-- Corrigir tabela Integration: renomear credentials → secretRef
ALTER TABLE "Integration" RENAME COLUMN "credentials" TO "secretRef";
```

**Arquivo afetado:** `prisma/dev.db`

### Passo 2: Criar tabela `_prisma_migrations` e registrar a migration

Para que o Prisma reconheça a migration como aplicada:

```sql
CREATE TABLE IF NOT EXISTS "_prisma_migrations" (
    "id" TEXT NOT NULL PRIMARY KEY,
    "checksum" TEXT NOT NULL,
    "finished_at" DATETIME,
    "migration_name" TEXT NOT NULL,
    "logs" TEXT,
    "rolled_back_at" DATETIME,
    "started_at" DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    "applied_steps_count" INTEGER NOT NULL DEFAULT 0
);

INSERT INTO "_prisma_migrations" ("id", "checksum", "finished_at", "migration_name", "started_at", "applied_steps_count")
VALUES (
    'manual-fix-001',
    'skip',
    CURRENT_TIMESTAMP,
    '20260624084500_data_persistence_foundation',
    CURRENT_TIMESTAMP,

```

**Arquivo afetado:** `prisma/dev.db`

## Riscos

- **Dados existentes:** As colunas adicionadas são `NULL` por padrão. Leads existentes terão `deletedAt = NULL` e `externalId = NULL`, o cual é compatível com o comportamento esperado (não deletados, sem external ID).
- **Migration formal:** O Prisma não terá registro histórico completo. Para futuras migrações, considerar resetar o banco com `prisma migrate reset` quando seguro.
- **Prisma engine:** O problema do engine que impediu `prisma migrate deploy` continua aberto. Esta correção é um bypass via SQL direto.
